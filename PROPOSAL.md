# Ultimate Visibility ETL Framework Proposal

## Overview
This document outlines the proposed Extract, Transform, Load (ETL) framework designed to achieve "Ultimate Visibility" across our application. The framework leverages a modern observability stack comprising Vector, Prometheus, Loki, and Grafana to collect, process, and visualize critical system metrics and logs.

## Infrastructure Components
*   **Vector:** Acts as the high-performance log router and processor. It reads raw logs (e.g., `workspace/logs/etl/system_metrics.jsonl`), transforms them into structured metrics using `log_to_metric` (prefixed with `etl_`), and extracts explicit tags before forwarding data to sinks.
*   **Prometheus:** Serves as the time-series database. It ingests the metrics generated by Vector and evaluates alerting rules (e.g., using `deriv()` for gauges and `sum(rate())/sum(rate())` for histograms) to detect anomalies.
*   **Loki:** Provides highly-available, multi-tenant log aggregation. Vector forwards raw logs to Loki, allowing deep correlation between metrics and log events without the overhead of full-text indexing.
*   **Grafana:** The visualization layer. It connects to Prometheus (`uid: Prometheus`) and Loki (`uid: Loki`) to display the Four Golden Signals in interactive, real-time dashboards.

## Top KPIs: The Four Golden Signals
To measure system health effectively, we will track the Four Golden Signals:

1.  **Latency (API Response Times):** The time it takes to service a request (e.g., `etl_http_request_duration_seconds_bucket`).
2.  **Traffic:** A measure of how much demand is being placed on the system (e.g., `etl_http_requests_total`).
3.  **Errors:** The rate of requests that fail (e.g., tracking `status=5xx` tags).
4.  **Saturation:** How "full" our service is, representing resource utilization (e.g., `etl_goroutines`, CPU, memory).

## Focus Area: API Response Times as a Go/No-Go Signal
Focusing on **API Response Times (Latency)** provides a critical, immediate indicator of system degradation when introducing new features.

When evaluating the rollout of a new feature (e.g., "Feature X"):
*   **The Go Signal:** If the feature is enabled in a canary release and the 95th/99th percentile API response times remain within our established Service Level Objectives (SLOs), it indicates the performance overhead is acceptable. We can proceed with a full rollout.
*   **The No-Go Signal:** If enabling the feature causes a statistically significant spike in latency—even if overall resource saturation remains low—it signals a potential bottleneck (e.g., inefficient database queries, synchronous blocking operations). This triggers an immediate rollback or "No-Go" decision until the performance regressions are addressed. By correlating latency metrics with Loki logs, engineering teams can rapidly pinpoint the exact subsystem causing the delay.
